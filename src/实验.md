你现在训练只用 8GB 显存，基本是正常现象，不是显卡“不能用满”。

看了你的代码，主要原因是：

1. 训练实际是单图跑
   [train_depth_gate.py](./train_depth_gate.py:260) 每次只取一个 `sample`，没有 DataLoader batch。`ACCUMULATION_STEPS=4` 只是梯度累积，等价于“算 4 次再更新一次参数”，但不是 4 张图同时放进显存。

2. SAM 和 YOLO 基本被冻结/不反传
   [pipeline.py](./pipeline.py:86) 里训练模式会冻结 SAM 主体；训练循环里 YOLO 和 `sam_predictor.set_image(rgb_np)` 都在 `torch.no_grad()` 下面。也就是说，最占显存的 SAM image encoder 不保存反向传播所需的激活。

3. 你真正训练的参数很少
   主要是 `DepthFeatureExtractor` 和 `mask_decoder.depth_gate`，这两个比 SAM ViT-H 主干小很多，所以不会吃满 24GB。

4. PyTorch 不会自动“用满显存”
   显存占用取决于模型参数、输入大小、batch size、是否保存梯度激活。能用 8GB 跑完，其实说明当前配置比较省显存。

如果你想提高显卡利用率/吞吐，可以考虑：

- 把训练改成真正的 batch，一次处理多张图，而不是只用 `accumulation_steps`。
- 把同一张图里的多个 bbox 合并成 batch 一次送进 `mask_decoder`，现在是逐 bbox 循环。
- 如果研究目标需要，可以解冻 SAM 的部分模块训练，但这会显著增加显存和过拟合风险。
- 增加输入分辨率或更复杂分支，不过这不一定提升效果。

一句话：你的 24GB 没用满，是因为当前训练策略很省显存；`ACCUMULATION_STEPS=4` 不会让显存变成 4 倍。真正要吃更多显存，需要“同时算更多东西”，例如真实 batch 或批量 bbox 解码。

**我们现在的问题在哪**

第一，**我们的方法还没有真正建模“amodal completion”**。
现在的 depth gate 只是把深度特征加到 SAM 的 image embedding 上：

```python
src = src + dense_prompt_embeddings
src = src + alpha * depth_delta
```

这不是一个显式的 visible-to-amodal 结构。HOM 的核心是先预测 visible，再基于 visible 去预测 amodal，再判断 occlusion。我们现在没有 visible branch、没有 occlusion branch、没有 invisible region loss，也没有“可见区域 -> 完整区域”的层级约束。因此模型很可能只学到“稍微改变 SAM mask 边界”，而不是学会补全遮挡区域。

第二，**YOLO 框和 amodal 目标之间天然冲突**。
YOLO 输出的大概率是可见物体框，或者至少是围绕可见证据的框；但我们训练目标是 amodal mask。SAM 又是强 prompt-following 模型，box prompt 会强烈限制它在框内分割。于是模型被要求做一件很拧巴的事：**在一个可见框提示下，输出超出可见证据的完整物体。** 这不是不可能，但必须有专门的 amodal 训练设计。现在只是对 decoder 加一个很弱的 depth residual，力量不够。

`depth_gate_alpha` 默认初值是 `0.005`，上限默认 `0.02`，而且默认不训练。也就是说深度分支对 SAM embedding 的影响非常小。这个设计适合“不要破坏 SAM”，但不适合“让 SAM 学会新的 amodal 能力”。如果结果接近原 SAM，这是预期内的。

你之前担心通道数，这里要分清楚：当前代码需要的是单通道 depth/disparity，因为 `DepthFeatureExtractor` 第一层是 `Conv2d(1,16,...)`。如果你拿官方 UOAIS-Net 的 3 通道 depth image 逻辑来对照，会觉得我们“通道数不对”；但按当前代码，单通道才是对的。真正的问题是：官方 UOAIS-Net 用固定 depth range 处理 RGB-D，而我们用每张图 percentile 归一化，这会改变深度物理含义。

第七，**现在和 HOM 的对比还不公平，也不容易赢**。
HOM 是完整训练的 RGB-D amodal instance segmentation 网络。我们现在冻结了 SAM 大部分，只训练一个很小的 depth branch 和 gate。优点是轻量，但要宣称超过 HOM，需要非常明确的实验优势，比如更强 zero-shot、少训练参数、更高 amodal IoU、更好真实场景泛化。否则单纯 mIoU 对比很难站住。

**接下来该怎么做**

我建议不要马上继续训练，而是分三阶段走。

**第一阶段：把 baseline 和实验目标钉死**

先明确我们要证明什么。最合理的论文卖点不是“我们全面超过 HOM”，而是：

> 利用 YOLO/SAM 的开放词汇与强泛化能力，通过轻量 RGB-D amodal adaptation，使 SAM 从 visible segmentation 扩展到 unseen object amodal segmentation，在部分指标或部分场景上超过 HOM。

也就是说，我们的优势应该是：

- 参数更少；
- 基于 SAM/YOLO，泛化更强；
- 不需要完整重新训练 Mask R-CNN/HOM；
- 在 OSD 或 UOAIS-Sim/real split 上 amodal/occlusion 指标更好或接近。

必须固定评估指标：

- amodal mask IoU / mIoU；
- visible mask IoU；
- occlusion region IoU；
- recall / precision；
- 按遮挡比例分层评估：low/mid/high occlusion；
- 按 depth quality 分层评估。

**第二阶段：先做干净的 oracle 训练验证**

在模型结构大改前，先回答一个关键问题：

> 如果给 SAM 正确 GT box，它有没有能力通过 depth 分支学出 amodal mask？

所以训练应该先做一个 oracle 版本：

- 不用 YOLO box；
- 直接用 GT amodal/visible bbox；
- 用正确 SAM postprocess 对齐 mask；
- 只评估 “GT box prompt + depth gate -> amodal mask”。

如果这个都学不好，说明 depth gate 结构不够。
如果这个能学好，再引入 YOLO box，研究真实 pipeline 下的性能损失。

**第三阶段：结构上要更接近 amodal 任务**

现在的 depth gate 不够像 amodal 模型。后续可以考虑三条路线：

1. 最小改动路线
   保留 SAM image encoder 和 prompt encoder，只改 mask decoder 输出：
   - visible mask head；
   - amodal mask head；
   - occlusion head；
   - loss 同时监督 visible/amodal/occluded。

2. HOM 对齐路线
   明确模仿 HOM 的层级：
   - box prompt feature；
   - visible prediction；
   - amodal prediction conditioned on visible feature；
   - occlusion prediction from visible/amodal relation。

3. SAM-friendly 路线
   不强迫 SAM 一步输出 amodal，而是两阶段：
   - SAM 先输出 visible；
   - depth + visible mask + box 再送一个 lightweight completion head；
   - completion head 输出 amodal mask。

我个人更倾向第三条。因为 SAM 本来就是强 visible/interactive segmentation 模型，硬让它的 mask decoder 直接变成 amodal decoder，可能会和预训练能力冲突。让 SAM 做它擅长的 visible，再加一个轻量 amodal completion head，逻辑更干净，也更容易解释。

**现在最优先排查/修正的顺序**

1. 先确认训练 mask 对齐是否错误。这个优先级最高。
2. 做 GT box oracle 训练，不要先用 YOLO box。
3. 使用官方风格 depth normalization 做对照：固定 `[2500,15000]`，不要每张图 percentile。
4. 加分层评估：遮挡比例、depth 边界质量、YOLO box 质量。
5. 检查 legacy GT 评估是否误用。UOAIS amodal 会重叠，应该用 per-instance `.npz` 评估，不要用单通道 amodal label map 做主要指标。
6. 再决定是否改结构：visible/amodal 双头，或者 SAM visible + completion head。

**一句话总结**

我们现在的问题是：**项目目标是 amodal completion，但当前实现只是 weak depth perturbation；训练目标用 amodal，prompt 却来自 visible/YOLO；评估想对标 HOM，但结构上没有 HOM 的 visible->amodal->occlusion 层级。**

所以接下来不要继续盲训 depth gate。应该先用 GT box oracle 证明“深度融合确实能让 SAM 学会 amodal”，再把 YOLO 放回 pipeline，最后再和 HOM 做正式对比。

> 在开始之前，我想就这一点YOLO 框和 amodal 目标之间天然冲突和你讨论一下，现在的问题是sam是在yolo之后的，那如果yolo漏检的话，sam就成了瞎子了，这种问题我们该怎么解决，还有就算我们先做了一个oracle版本，我们训练出的sam具备了非常强大的补全能力，但是如果yolo的bbox漏检，最终的效果依然很差，这该怎么解决，你不能做任何修改

对，这一点你抓得很准：**YOLO 漏检会直接决定整个 pipeline 的召回上限**。如果没有 box prompt，SAM 再强也不会主动知道该分哪个物体。这个问题不是训练一个更强 amodal-SAM 就能完全解决的，它是 pipeline 设计层面的瓶颈。

我会把问题拆成两层：

**第一层：YOLO 漏检**
YOLO 没给框，SAM 没入口，最终必漏。解决方向有几类：

1. **提高 proposal recall，而不是只追求检测 precision**
   在这个 pipeline 里，YOLO 的角色不应该是最终检测器，而应该是 proposal generator。也就是说宁愿多给一些框，也不要漏物体。可以降低 `conf_threshold`、放宽 NMS、保留更多候选框，然后让后面的 SAM/amodal head/评分模块去筛。

2. **用多个 proposal 来源补召回**
   不只依赖 YOLO。可以组合：
   - YOLO-World boxes
   - SAM automatic mask generator 产生的 visible proposals
   - depth edge / depth discontinuity proposals
   - class-agnostic object proposal 方法
   - sliding/window proposals 作为兜底

   对 unseen object amodal segmentation 来说，class-agnostic recall 很重要。UOAIS/HOM 本质上也是 class-agnostic unseen object，不应该完全绑死在 YOLO 分类检测能力上。

3. **从“检测框驱动”变成“proposal ensemble 驱动”**
   我们可以把 YOLO 框看作候选之一，而不是唯一入口。最终 pipeline 可以是：

   ```text
   RGB-D image
      -> proposal generator: YOLO + SAM AMG + depth proposals
      -> SAM visible masks
      -> amodal completion head
      -> mask scoring / NMS / merging
   ```

   这样即使 YOLO 漏了，SAM AMG 或 depth proposal 还有机会补回来。

4. **训练一个 objectness / mask scorer**
   如果 proposal 变多，假阳性也会多。所以需要后处理评分：
   - mask quality
   - depth consistency
   - visible/amodal plausibility
   - occlusion likelihood
   - duplicate suppression

   这样可以用“高召回 proposal + 后端筛选”替代“高精度 YOLO 一锤定音”。

**第二层：YOLO 框和 amodal mask 不一致**

即使 YOLO 没漏，YOLO box 通常也是可见区域附近的框，而 amodal mask 可能超出这个框。这个冲突可以这样缓解：

1. **训练时不要只用 tight visible box**
   可以用 GT amodal box、visible box、YOLO-like noisy box、expanded box 混合训练，让模型学会从不同 prompt 下补全。

2. **推理时扩张 YOLO box**
   如果直接把 YOLO tight box 给 SAM，它会倾向在框内分割。可以对 box 做比例扩张，比如 1.1x、1.25x、1.5x 多尺度 box，然后让 amodal head 选择最合理结果。

3. **让 box 只表示“物体中心证据”，不要表示完整边界**
   论文表述上可以把 YOLO box 定义成 visible evidence prompt，而不是完整 object extent。后续 amodal module 的任务就是从 visible evidence 推断 complete extent。

4. **把 visible mask 作为中间状态**
   更合理的结构是：

   ```text
   YOLO/SAM proposal -> visible mask -> depth-aware amodal completion
   ```

   而不是：

   ```text
   YOLO box -> SAM mask decoder 直接输出 amodal mask
   ```

   因为 visible mask 可以告诉模型“当前可见部分在哪里”，depth 再帮助判断遮挡边界和补全方向。

**那 oracle 版本还有没有意义？**

有意义，但它不是最终方案。oracle 版本的作用不是解决 YOLO 漏检，而是回答一个更基础的问题：

> 在 proposal 完美的情况下，我们的 amodal-SAM 是否真的具备补全能力？

如果 oracle 都不好，说明补全模块本身不成立。
如果 oracle 很好，但 YOLO pipeline 很差，说明瓶颈在 proposal recall。
这两个结论对应完全不同的改进方向。

所以我建议把实验拆成三个上限：

```text
1. GT box + SAM/amodal head
   测补全模块能力上限

2. YOLO box + SAM/amodal head
   测真实 YOLO pipeline 性能

3. YOLO + high-recall proposals + SAM/amodal head
   测完整系统潜力
```

这样我们能清楚知道差距来自哪里。

**最终要和 HOM 对比时，我建议这样定位**

不要说“我们只靠 YOLO 检测就解决 unseen object amodal segmentation”。这会被 YOLO 漏检问题卡住。

更稳的说法是：

> 我们提出一个 proposal-driven RGB-D amodal segmentation framework，以 YOLO/SAM 作为开放词汇与强泛化 proposal source，并通过 depth-aware amodal completion head 补全不可见区域。

这样 YOLO 只是 proposal 来源之一，而不是系统唯一眼睛。

**我认为最现实的路线**

先做：

```text
GT box oracle
```

验证补全能力。

然后做：

```text
YOLO-only proposal
```

量化 YOLO 漏检造成的 recall 损失。

再做：

```text
YOLO + SAM AMG/depth proposal
```

补召回。

最终系统应该不是“YOLO 后接 SAM”这么单线，而是：

```text
RGB-D
 -> high-recall proposal generator
 -> SAM visible segmentation
 -> depth-aware amodal completion
 -> scoring/NMS
 -> final amodal instances
```

一句话：**YOLO 漏检不能靠训练 SAM 来解决，必须靠 proposal recall 机制解决；oracle 训练解决的是补全模块是否成立，proposal ensemble 解决的是系统召回上限。**

> 我觉得oracle测试可以看成一个训练的机会，如果说我们按照你上面所说的对yolo和sam分别进行了改造，那接下来还要训练，如果我把oracle用到的完美的proposal作为训练的原料分别输入给yolo和sam，那是不是可以降低yolo的漏检和提高sam的补全能力，然后最后再把训练完成的两部分合起来验证非模态补全能力

你的想法方向是对的，但要稍微拆清楚：**oracle proposal 可以非常适合训练 SAM/amodal 补全模块；但它不能直接“输入给 YOLO”来降低漏检，除非你把它转化成 YOLO 的检测监督。**

换句话说：

```text
GT proposal / oracle box
  -> 非常适合训练 SAM 后面的 amodal completion
  -> 也可以用来训练/微调 YOLO，但方式不是作为输入，而是作为检测标签
```

**对 SAM/amodal 部分**

这个很成立。你可以把完美 proposal 当训练原料：

```text
GT visible box / GT amodal box / expanded GT box
    + RGB
    + depth
    -> SAM/amodal module
    -> amodal mask loss
```

这样做的价值很大：

- 去掉 YOLO 漏检和错框噪声；
- 让模型先专心学习“给定一个物体提示，如何补全 amodal mask”；
- 可以训练多种 prompt 鲁棒性：tight visible box、expanded visible box、amodal box、jitter box；
- 训练完成后再接真实 YOLO proposal，看性能掉多少。

这个就是我说 oracle training 的核心价值。

**对 YOLO 部分**

YOLO 不能直接把 oracle proposal 当“输入”。YOLO 输入仍然是图像，输出是框。
但你可以用 oracle proposal 生成检测标签，去微调 YOLO：

```text
RGB image -> YOLO -> predict box
GT visible/amodal boxes -> detection loss
```

这里有一个关键选择：**YOLO 到底要学 visible box 还是 amodal box？**

我建议不要让 YOLO 学 amodal box。原因是 YOLO 从 RGB 里只能看到 visible evidence，让它直接预测完整 amodal extent 会很难，而且容易不稳定。

更合理的是：

```text
YOLO 学 high-recall visible/objectness box
SAM/amodal module 学 visible -> amodal
```

也就是：

- YOLO 目标：尽量别漏，框住可见物体；
- SAM visible branch：得到可见 mask；
- depth-aware completion：补全 amodal mask。

**最终训练可以分成两条线**

1. Proposal 训练线，也就是 YOLO/检测器

目标不是最高 mAP，而是高 recall。

训练数据：

```text
RGB image
GT visible bbox 或 visible mask 外接框
```

训练策略：

- 降低漏检；
- class-agnostic objectness；
- 可以用 UOAIS 的所有 object 类都合成一个 `object` 类；
- 评价重点用 proposal recall：GT instance 有多少被至少一个 proposal 覆盖。

指标：

```text
Recall@IoU 0.3 / 0.5
Average proposals per image
Miss rate
```

2. Amodal completion 训练线，也就是 SAM/depth/amodal head

训练数据：

```text
RGB + depth + oracle proposal
target = amodal mask
辅助 target = visible mask / occluded mask
```

训练策略：

- 先用 GT proposal；
- 再加入 jitter proposal；
- 最后加入 YOLO proposal 做 domain adaptation。

指标：

```text
Amodal mIoU
Visible mIoU
Occlusion mIoU
```

**最后合起来验证**

最终 pipeline 应该这样验证：

```text
RGB-D
 -> trained high-recall YOLO/proposal generator
 -> SAM visible segmentation
 -> trained amodal completion module
 -> final amodal masks
```

然后你要做一个很重要的 ablation 表：

```text
A. GT box + amodal module
B. jittered GT box + amodal module
C. YOLO box + amodal module
D. trained YOLO box + amodal module
E. trained YOLO + extra proposals + amodal module
```

这个表能回答：

- 补全模块本身上限是多少；
- YOLO 框带来多少性能损失；
- 微调 YOLO 后召回提升多少；
- 整体系统能否超过 HOM。

**一个关键提醒**

不要把 oracle 训练后的结果直接当最终结果。因为 reviewer 很容易问：

> 你用了 GT box，那这不是检测任务被绕过了吗？

所以 oracle 只能作为上限实验和训练阶段，最终主结果必须用自动 proposal。

但 oracle training 很有价值，甚至应该成为第一阶段训练：

```text
Stage 1: GT proposal 训练 amodal completion
Stage 2: jitter proposal 增强鲁棒性
Stage 3: YOLO proposal 微调/适配
Stage 4: end-to-end system evaluation
```

一句话：**oracle proposal 可以用来训练 SAM/amodal 补全能力；也可以转成 YOLO 的检测标签去降低漏检；但最终必须用自动 proposal 验证，oracle 只能做上限和训练引导。**

> 我们的项目等于说现在是夹在中间的，前面有sam和yolo已有的模型约束（提示词冲突，权重需要冻结等问题）后面有我们项目最终想要对标的论文Unseen Object Amodal Instance Segmentation via Hierarchical Occlusion Modeling的各种限制，loss的计算，各个实验结果的数据对比等等，我现在需要你告诉我往前也就是sam和yolo我们对其进行改造需要注意什么往后也就是这篇论文我们又需要注意什么

对，你这个“夹在中间”的描述非常准确。我们不是从零做一个 amodal 网络，而是在 **YOLO/SAM 既有模型约束** 和 **HOM/UOAIS 论文评测范式** 之间搭桥。这个桥如果两头没对齐，就会出现：前面模型能力没释放，后面指标又对不上，最后实验很难讲清楚。

我建议你把项目拆成两个方向同时约束。

**往前：改造 YOLO 和 SAM 要注意什么**

第一，YOLO 不应该被定义成最终检测器，而应该是 high-recall proposal generator。
我们的目标不是 YOLO mAP 好看，而是别漏掉 UOAIS 里的 unseen objects。YOLO 的指标应该先看 proposal recall：

```text
GT instance 是否至少被一个 proposal 覆盖
Recall@IoU 0.3 / 0.5
每张图平均 proposal 数
漏检率
```

所以改 YOLO 时要注意：

- 优先提高 recall，不要过早追求 precision。
- 可以 class-agnostic 训练，把所有物体当 object。
- 训练目标优先用 visible bbox，而不是 amodal bbox。
- 推理时可以降低 conf、放宽 NMS、保留 top-k proposals。
- YOLO 框最好做多尺度扩张，因为 tight visible box 会限制 amodal 补全。
- 不要让最终系统只依赖 YOLO，最好准备 SAM AMG/depth proposal 作为召回兜底。

第二，SAM 的强项是 visible segmentation，不要强迫它直接变成完整 amodal 网络。
SAM 的预训练先验非常强，但它的 prompt-following 也很强。给它一个 tight box，它天然倾向输出框内可见区域。改 SAM 时要注意：

- 尽量冻结 image encoder，先不要破坏 SAM 的视觉表征。
- prompt encoder 可以先冻结，因为 prompt 语义稳定。
- mask decoder 可以改，但最好只加小模块，不要整块重训。
- 训练时要包含多种 prompt：GT visible box、expanded box、jitter box、YOLO-like noisy box。
- 不要只训练 GT amodal mask，最好同时监督 visible mask 和 occluded mask。
- 深度融合不要直接覆盖 SAM dense prompt，应当 residual/gated/side branch。
- 推理路径必须确认真的用上 depth 分支，不要出现“传了 depth 但没接入 decoder”的假深度。
- 低分辨率 mask 回原图做 loss 时，必须走 SAM 官方 postprocess 流程，避免 padding/resize 错位。

第三，YOLO 和 SAM 中间要有“proposal 语义转换”。
YOLO 框不是 amodal 框，它只是 visible evidence prompt。系统叙述上应该明确：

```text
YOLO proposal = visible evidence
SAM visible mask = observed object support
Depth-aware completion = amodal inference
```

这样可以避免“为什么 YOLO 框不覆盖完整物体，却要求 SAM 输出完整物体”的逻辑冲突。

第四，训练要分阶段。
不要一开始就 YOLO+SAM+depth 全串起来训。建议：

```text
Stage 1: GT box oracle 训练 amodal completion 上限
Stage 2: jitter/expanded box 训练 prompt 鲁棒性
Stage 3: YOLO proposal 适配真实 pipeline
Stage 4: high-recall proposal ensemble 提升召回
```

**往后：对标 HOM/UOAIS 要注意什么**

第一，任务定义必须一致。
HOM/UOAIS 做的是 unseen object amodal instance segmentation，通常关注：

```text
visible mask
amodal mask
occluded region
instance-level matching
```

所以我们不能只报普通 SAM visible mask 的 IoU，也不能只报单张图有没有好看的补全。必须报 instance-level 指标。

第二，GT 格式要用 per-instance mask stack。
UOAIS 的 amodal mask 可能重叠，单通道 instance id map 会丢信息。你现在已经有 `exported_instance_gt_full/*.npz`，这个方向是对的。主评估不要依赖 legacy 单通道 `gt_amodal.npy`，否则指标可能不可信。

第三，loss 要和论文目标对应。
HOM 的核心是层级遮挡建模：visible、amodal、occlusion 有关系。我们至少要有类似的监督：

```text
L_visible
L_amodal
L_occlusion
L_consistency: visible ⊆ amodal
L_occlusion_consistency: occluded = amodal - visible
```

如果只用 amodal BCE/Dice，模型可能学不到“哪里是可见，哪里是补全”。

第四，指标要能拆出问题来源。
最终和 HOM 对比时，建议至少有：

```text
mIoU amodal
mIoU visible
mIoU occlusion
precision / recall
按遮挡比例分层
按 proposal 类型分层
按 depth quality 分层
```

尤其是 occlusion region IoU，这是证明“非模态补全能力”的关键。只看 amodal mIoU，提升可能被可见区域淹没。

第五，实验表要区分 oracle 和 automatic。
HOM 是自动系统，我们最终主结果也必须是自动 proposal。Oracle 只能作为上限实验：

```text
GT box + our completion  -> upper bound
YOLO box + our completion -> real pipeline
YOLO+extra proposals -> improved pipeline
HOM -> paper baseline
```

不能拿 GT box oracle 主结果去和 HOM 自动结果比，否则不公平。

第六，数据处理要尽量贴近 UOAIS 官方。
UOAIS/HOM 是 RGB-D 方法，depth 处理会影响结果。要注意：

更合理的训练路线我建议是这样：

```text
Stage 0: 数据与评估协议固定
Stage 1: SAM/amodal 模块用 oracle proposal 训练补全上限
Stage 2: SAM/amodal 模块用 jitter / expanded / YOLO-like proposal 训练鲁棒性
Stage 3: YOLO 用 GT visible boxes 训练 high-recall proposal
Stage 4: YOLO proposals 接入 SAM/amodal 模块做适配训练
Stage 5: 加入额外 proposal 来源，降低 YOLO 漏检瓶颈
Stage 6: 完整系统评估，对标 HOM
```

你说的核心逻辑是成立的：

- **GT 不存在漏检**，所以用 oracle proposal 训练 SAM 补全能力，不会被 YOLO 漏检误导。
- 同一份 GT 可以转成检测标签训练 YOLO，提高 proposal recall。
- YOLO 漏检不能只靠训练 YOLO 解决，还要引入其他 proposal source 兜底。

我会特别强调几点。

**1. YOLO 和 SAM 的训练目标要分开**

YOLO 的目标不是 amodal completion，而是：

```text
尽可能找全物体可见证据
```

所以 YOLO 更适合训练成 class-agnostic high-recall objectness detector。标签建议优先用：

```text
visible bbox
```

而不是 amodal bbox。因为 YOLO 从 RGB 里看到的是可见部分，让它直接学 amodal extent 会很虚。

SAM/amodal 模块的目标才是：

```text
visible evidence + RGB-D -> amodal mask
```

它可以用：

```text
GT visible bbox / expanded bbox / jitter bbox / YOLO-like bbox
```

作为 prompt，target 用 amodal mask，同时最好辅助监督 visible mask 和 occlusion mask。

**2. oracle 是上限训练，不是最终验证**

你说“采取 oracle 来训练”是对的，但最终验证不能只用 oracle。最终应该有三条结果：

```text
GT box + SAM/amodal       补全能力上限
YOLO box + SAM/amodal     真实 YOLO pipeline
YOLO+extra proposals      完整系统
```

这样才能解释清楚：
如果 GT box 好、YOLO box 差，问题在 proposal。
如果 GT box 都差，问题在 amodal module。
如果 extra proposals 后提升，说明漏检确实是瓶颈，并且我们有解决方案。

**3. YOLO 漏检需要多方案兜底**

你这个判断也对。不要只靠 YOLO 微调。可以引入：

```text
低阈值 YOLO top-k proposals
box expansion
SAM automatic mask generator proposals
depth discontinuity proposals
class-agnostic proposal network
multi-scale inference
proposal ensemble + NMS
```

核心策略是：**前端宁愿多给框，后端再筛掉假阳性。**

这就要求后面加一个 scoring/filtering 机制，比如：

- SAM mask quality；
- depth consistency；
- visible/amodal consistency；
- duplicate NMS；
- objectness score；
- occlusion plausibility。

**4. SAM 补全训练不能只用完美 box**

如果只用完美 oracle box 训练，模型可能一接 YOLO box 就掉性能。
所以 oracle 后面必须加 prompt augmentation：

```text
GT visible box
expanded visible box
GT amodal box
random jitter box
partial/tight box
YOLO-predicted box
```

这样 SAM/amodal 模块才知道真实推理时 prompt 会不准。

**5. 我会小改你的表述**

你现在说：

> 在完成对于 sam 和 yolo 的改造之后，把他们拆分为两个部分分开训练。

我建议改成：

> 先拆成 proposal learning 和 amodal completion learning 两个任务，分别定义上限和训练协议，再决定 YOLO/SAM 需要怎么改造。

这样更稳。因为如果先改造，再发现训练目标不对，会返工。

**最终判断**

你的方案是对的，但我会改成：

```text
SAM/amodal:
先 oracle 训练补全能力
再 jitter/YOLO proposal 训练鲁棒性

YOLO:
用 GT visible bbox 训练 high-recall detector
再配合 extra proposals 解决漏检

完整系统:
trained YOLO/proposals + trained SAM/amodal completion
自动推理评估
```

一句话：**GT/oracle 数据既可以作为 SAM 补全训练的干净入口，也可以作为 YOLO 高召回训练的检测标签；但二者目标不同，SAM 学补全，YOLO 学别漏，最终还要靠 proposal ensemble 降低漏检对系统上限的限制。**
